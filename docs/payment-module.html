Payment Module

Technical deep-dive for backend developers — architecture, code flows, and design decisions.

Overview

The Payment module handles all money movement in the system. It supports two checkout pathways controlled by a Strategy Pattern:

in_person offline

Cash or physical terminal. The system writes a pending_cash ledger row and immediately marks the order as paid — synchronous, no redirect.

online async

Bank gateway redirect (Zarinpal). System creates an initiated payment row, returns a redirect URL, and waits for the bank to call back via a public GET endpoint.

Module boundary rule: Payment only crosses into Order via OrderManagerInterface (a contract). It never imports Modules\Order\Domain\Models\Order. All other modules are off-limits.

Domain Layer

Enums

PHP 8.1 backed enums so the string values are stored in the DB and you can always PaymentStatus::from($row->status) without a map.

enum PaymentMethodType: string
{
    case ONLINE    = 'online';
    case IN_PERSON = 'in_person';
}

enum PaymentStatus: string
{
    case INITIATED    = 'initiated';     // online: payment request sent, waiting callback
    case CAPTURED     = 'captured';      // online: gateway verify succeeded
    case FAILED       = 'failed';        // cancelled by user or rejected by gateway
    case REFUNDED     = 'refunded';      // reserved for future refund flow
    case PENDING_CASH = 'pending_cash';  // in_person: cash collected offline
}

DTOs

All DTOs are immutable — every property is public readonly. They cross module boundaries; Eloquent models never do.

// GatewayRedirectDTO — returned by requestPayment()
class GatewayRedirectDTO
{
    public function __construct(
        public readonly string $redirectUrl,         // where to send the user's browser
        public readonly string $transactionReference, // Zarinpal Authority token
    ) {}
}
// PaymentDTO — full ledger row for API responses
class PaymentDTO
{
    public function __construct(
        public readonly int               $id,
        public readonly int               $orderId,
        public readonly PaymentMethodType $methodType,
        public readonly ?string           $gateway,
        public readonly ?string           $transactionReference,
        public readonly int               $amount,         // always in rials
        public readonly PaymentStatus     $status,
        public readonly ?array            $gatewayResponse,
        public readonly Carbon            $createdAt,
    ) {}

    public static function fromModel(Payment $payment): self
    {
        return new self(
            id:                   $payment->id,
            orderId:              $payment->order_id,
            methodType:           PaymentMethodType::from($payment->method_type),
            gateway:              $payment->gateway,
            transactionReference: $payment->transaction_reference,
            amount:               $payment->amount,
            status:               PaymentStatus::from($payment->status),
            gatewayResponse:      $payment->gateway_response,
            createdAt:            Carbon::parse($payment->created_at),
        );
    }
}

Contracts (Public Interfaces)

These two interfaces are the only things other modules or the HTTP layer may depend on from this module.

    public function initializePayment(int $orderId, string $methodType, ?string $gateway = null): array;

Database Schema

ColumnTypeNotes
idBIGINT UNSIGNED PKAuto-increment
order_idBIGINT UNSIGNED, indexedLoose reference — no DB foreign key. Index gives query performance without coupling the tables. Follows the same pattern as media_id in other modules.
method_typeVARCHARonline or in_person
gatewayVARCHAR nullablezarinpal, mock, or null for in_person
transaction_referenceVARCHAR UNIQUE nullableStores Zarinpal Authority on init; replaced by permanent ref_id on capture. UNIQUE prevents duplicate captures from double-delivery of callbacks.
amountINT (not DECIMAL)Always in rials — the Cents Rule. Never a float.

Why no DB foreign key to orders? Adding REFERENCES orders(id) at the DB layer would couple the two tables permanently, making it impossible to extract either module into a microservice without a schema migration. The application layer enforces the relationship via OrderManagerInterface::findOrder().

Gateway Strategy Pattern

The Strategy Pattern lets you swap payment gateways without touching business logic. Three pieces work together:

// How it connects:
// InitializePaymentAction    → calls PaymentGatewayFactory::make('zarinpal')
// PaymentGatewayFactory      → resolves ZarinpalGatewayDriver (implements interface)
// InitializePaymentAction    → calls $driver->requestPayment(...)
// In tests: factory resolves → MockGatewayDriver (same interface, no HTTP)
// The action never knows which concrete driver it's using

PaymentGatewayDriverInterface

requestPayment() — called during checkout. Creates a payment session at the gateway and returns a redirect URL plus an authority token.

verifyPayment() — called on callback. Sends the authority back to the bank to confirm the transaction really happened and returns a permanent reference ID.

ZarinpalGatewayDriver

Production driver for Zarinpal REST API v4. Uses Laravel's Http facade.

class ZarinpalGatewayDriver implements PaymentGatewayDriverInterface
{
    private const REQUEST_URL   = 'https://payment.zarinpal.com/pg/v4/payment/request.json';

    private const VERIFY_URL    = 'https://payment.zarinpal.com/pg/v4/payment/verify.json';

    private const START_PAY_URL = 'https://payment.zarinpal.com/pg/StartPay/';

    public function requestPayment(int $orderId, int $amountInCents, ...): GatewayRedirectDTO
    {
        $response = Http::post(self::REQUEST_URL, [
            'merchant_id'  => config('payment.gateways.zarinpal.merchant_id'),
            'amount'       => $amountInCents,   // rials, sent as-is
            'callback_url' => $callbackUrl,
            'description'  => "Payment for order #{$orderId}",
            'metadata'     => $customerMetadata ?? [],
        ]);

        $data = $response->json();

        if (($data['data']['code'] ?? null) !== 100) {
            throw new RuntimeException('Zarinpal request failed: ' . json_encode($data['errors'] ?? $data));
        }

        $authority = $data['data']['authority'];

        return new GatewayRedirectDTO(
            redirectUrl:          self::START_PAY_URL . $authority,
            transactionReference: $authority,  // stored in payments.transaction_reference
        );
    }

    public function verifyPayment(int $amountInCents, string $authority): array
    {
        $data = Http::post(self::VERIFY_URL, [
            'merchant_id' => config('payment.gateways.zarinpal.merchant_id'),
            'amount'      => $amountInCents,
            'authority'   => $authority,
        ])->json();

        $code    = $data['data']['code'] ?? null;
        $success = in_array($code, [100, 101], true); // 100=new, 101=already verified

        return [
            'success'      => $success,
            'reference_id' => $success ? (string)($data['data']['ref_id'] ?? '') : null,
            'raw_response' => $data,  // saved to gateway_response column
        ];
    }
}
Zarinpal response codes: 100 = new successful capture. 101 = already verified before (idempotent re-check from bank side). Anything else = error. The authority is a temporary session token; the ref_id is the permanent bank transaction reference.

MockGatewayDriver

A test double that implements the same interface but makes zero HTTP calls. Configure it before the test request hits the app.

class MockGatewayDriver implements PaymentGatewayDriverInterface
{
    public bool   $shouldVerifySucceed = true;
    public string $fakeAuthority        = 'MOCK-AUTHORITY-0000000001';
    public string $fakeRefId            = 'MOCK-REF-12345';

    public function requestPayment(...): GatewayRedirectDTO
    {
        return new GatewayRedirectDTO(
            redirectUrl:          'https://mock-gateway.test/pay/' . $this->fakeAuthority,
            transactionReference: $this->fakeAuthority,
        );
    }

    public function verifyPayment(int $amount, string $authority): array
    {
        if (! $this->shouldVerifySucceed) {
            return ['success' => false, 'reference_id' => null,
                    'raw_response' => ['data' => ['code' => -22]]];
        }
        return ['success' => true, 'reference_id' => $this->fakeRefId,
                'raw_response' => ['data' => ['code' => 100, 'ref_id' => $this->fakeRefId]]];
    }
}

// In tests — override the singleton in the container before the request:
$driver = new MockGatewayDriver();
$driver->shouldVerifySucceed = false;                         // simulate rejection
$this->app->instance(MockGatewayDriver::class, $driver);

PaymentGatewayFactory

A singleton that maps string names to driver classes and resolves them from the container.

class PaymentGatewayFactory
{
    private array $drivers = [
        'zarinpal' => ZarinpalGatewayDriver::class,
        'mock'     => MockGatewayDriver::class,
    ];

    public function make(string $gateway): PaymentGatewayDriverInterface
    {
        if (! isset($this->drivers[$gateway])) {
            throw new InvalidArgumentException("Unknown gateway driver: [{$gateway}]");
        }
        return app($this->drivers[$gateway]); // app() lets tests override via $app->instance()
    }

    public function register(string $name, string $class): void
    {
        $this->drivers[$name] = $class; // extensibility: e.g. a PayPalServiceProvider can add 'paypal'
    }
}
Why app() instead of new $class()? Using app() allows the test to bind a configured MockGatewayDriver instance via $this->app->instance(). If you used new, the test override would be bypassed entirely.

Application Actions

Actions are the business logic layer. One handle() method each, depend on Contracts never concrete classes, and wrap all multi-step DB writes in DB::transaction().

InitializePaymentAction

The main checkout logic. Branches on method_type and handles both payment pathways.

class InitializePaymentAction
{
    public function __construct(
        private readonly OrderManagerInterface  $orderManager,
        private readonly PaymentGatewayFactory $gatewayFactory,
    ) {}

    public function handle(int $orderId, string $methodType, ?string $gateway = null): array
    {
        // 1. Load order via contract — zero Order model imports
        $order = $this->orderManager->findOrder($orderId);
        if ($order === null) { abort(404, 'Order not found.'); }

        $method = PaymentMethodType::from($methodType);

        // ── Branch A: in_person ──────────────────────────────────────────
        if ($method === PaymentMethodType::IN_PERSON) {
            return DB::transaction(function () use ($orderId, $order) {
                $ref     = 'CASH-' . uniqid();
                $payment = Payment::create([
                    'order_id'              => $orderId,
                    'method_type'           => PaymentMethodType::IN_PERSON->value,
                    'amount'                => $order->totalAmount,  // int rials from OrderDTO
                    'status'                => PaymentStatus::PENDING_CASH->value,
                    'transaction_reference' => $ref,
                ]);
                $this->orderManager->markAsPaid($orderId, $ref);
                return ['type' => 'in_person', 'payment_id' => $payment->id,
                        'status' => PaymentStatus::PENDING_CASH->value, 'redirect_url' => null];
            });
        }

        // ── Branch B: online ─────────────────────────────────────────────
        $resolvedGateway = $gateway ?? config('payment.default_gateway');
        $driver          = $this->gatewayFactory->make($resolvedGateway);
        $redirectDto     = $driver->requestPayment(
            orderId:       $orderId,
            amountInCents: $order->totalAmount,
            callbackUrl:   url('/api/v1/payments/zarinpal/callback'),
        );

        $payment = Payment::create([
            'order_id'              => $orderId,
            'method_type'           => PaymentMethodType::ONLINE->value,
            'gateway'               => $resolvedGateway,
            'amount'                => $order->totalAmount,
            'status'                => PaymentStatus::INITIATED->value,
            'transaction_reference' => $redirectDto->transactionReference, // Authority token
        ]);

        return ['type' => 'online', 'payment_id' => $payment->id,
                'status' => PaymentStatus::INITIATED->value,
                'redirect_url' => $redirectDto->redirectUrl];
    }
}
Why transaction scope differs: The in_person branch wraps both writes in DB::transaction() — if markAsPaid throws, the payment row rolls back too. The online branch does NOT use a transaction around the gateway call because you cannot wrap external HTTP I/O in a DB transaction meaningfully. If the Zarinpal call throws, the Payment row is never created — no rollback needed.

HandleZarinpalCallbackAction

Called when the user returns from the bank. Zarinpal sends Status and Authority as GET query parameters.

public function handle(string $status, string $authority): array
{
    // 1. Find payment by the authority we stored on initialize
    $payment = Payment::where('transaction_reference', $authority)->first();
    if ($payment === null) {
        return ['success' => false, 'message' => 'Payment record not found.'];
    }

    // 2. User cancelled at bank — skip verify entirely
    if ($status !== 'OK') {
        $payment->update(['status' => PaymentStatus::FAILED->value]);
        return ['success' => false, 'message' => 'Payment was cancelled by the user.'];
    }

    // 3. Idempotency guard — callback may be delivered multiple times
    if ($payment->status === PaymentStatus::CAPTURED->value) {
        return ['success' => true, 'message' => 'Payment already captured.',
                'payment_id' => $payment->id];
    }

    // 4. Verify with the same gateway that created this payment
    $driver = $this->gatewayFactory->make($payment->gateway ?? 'zarinpal');
    $result = $driver->verifyPayment($payment->amount, $authority);

    // 5. Write outcome atomically
    return DB::transaction(function () use ($payment, $result) {
        if (! $result['success']) {
            $payment->update(['status' => PaymentStatus::FAILED->value,
                              'gateway_response' => $result['raw_response']]);
            return ['success' => false, 'message' => 'Payment verification failed.'];
        }

        // Overwrite Authority with permanent ref_id
        $payment->update([
            'status'                => PaymentStatus::CAPTURED->value,
            'transaction_reference' => $result['reference_id'], // permanent bank ref
            'gateway_response'      => $result['raw_response'],
        ]);
        $this->orderManager->markAsPaid($payment->order_id, (string) $result['reference_id']);
        return ['success' => true, 'payment_id' => $payment->id,
                'reference_id' => $result['reference_id']];
    });
}
The idempotency guard is critical. Banks and network retries can deliver the same callback multiple times. Without the guard, you'd call verifyPayment() twice — Zarinpal returns code 101 on the second call (already verified), your code would call markAsPaid twice, and write duplicate updates. The guard short-circuits on status === 'captured' before any of that happens.
Why does transaction_reference change on capture? On initialize it stores the temporary Authority token (used to find the row when the callback arrives). On capture it is overwritten with the permanent ref_id from the bank. This is the field you surface to customers for dispute resolution. The UNIQUE constraint ensures you can never accidentally create two payments with the same authority or the same ref_id.

PaymentController

Thin controller — no business logic. Delegates entirely to the Action/Manager and formats the response.

class PaymentController extends Controller
{
    public function __construct(
        private readonly PaymentManagerInterface       $manager,
        private readonly HandleZarinpalCallbackAction  $handleCallback,
    ) {}

    public function initialize(InitializePaymentRequest $request): JsonResponse
    {
        // FormRequest already ran authorize() + rules() before this executes
        $result = $this->manager->initializePayment(
            orderId:    (int) $request->input('order_id'),
            methodType: $request->input('method_type'),
            gateway:    $request->input('gateway'),
        );
        return response()->json($result);
    }

    public function zarinpalCallback(Request $request): JsonResponse
    {
        $status    = (string) $request->query('Status', '');
        $authority = (string) $request->query('Authority', '');

        if (empty($authority)) {
            return response()->json(
                ['success' => false, 'message' => 'Missing Authority parameter.'], 400
            );
        }

        $result     = $this->handleCallback->handle($status, $authority);
        $httpStatus = $result['success'] ? 200 : 422;
        return response()->json($result, $httpStatus);
    }
}

InitializePaymentRequest

authorize() runs before rules(), so unauthorized users get 403, not 422. Permission-based, not role-based — any user granted payment.create can pay.

public function authorize(): bool
{
    return (bool) $this->user()?->can('payment.create');
}

public function rules(): array
{
    return [
        'order_id'    => ['required', 'integer'],
        'method_type' => ['required', 'string', 'in:online,in_person'],
        'gateway'     => ['nullable', 'string'],
    ];
}

Complete Request Flows

In-Person Checkout Flow

1
POST /api/v1/payments/initialize

Body: {"order_id":42,"method_type":"in_person"}. Header: Authorization: Bearer {token}

2
auth:sanctum middleware

Resolves user from Sanctum token. Returns 401 if missing, 429 if rate limit hit.

3
InitializePaymentRequest::authorize()

$user->can('payment.create') — returns 403 if false. Runs before rules.

4
InitializePaymentRequest::rules()

Validates order_id (required int), method_type (in:online,in_person). Returns 422 on failure.

5
InitializePaymentAction — in_person branch — DB::transaction()

Calls orderManager->findOrder(42). Gets OrderDTO{totalAmount:150000}. Creates payment row (pending_cash). Calls orderManager->markAsPaid(42, 'CASH-...'). Both writes in one transaction — either both commit or both roll back.

6
Response 200

{"type":"in_person","payment_id":1,"status":"pending_cash","redirect_url":null}

Online Payment Flow — Initialize

1
POST /api/v1/payments/initialize

Body: {"order_id":42,"method_type":"online","gateway":"zarinpal"}

2
InitializePaymentAction — online branch

Resolves gateway name → factory → ZarinpalGatewayDriver.

3
ZarinpalGatewayDriver::requestPayment()

HTTP POST to Zarinpal. Gets authority="A000000..." and code 100. Returns GatewayRedirectDTO.

4
Write payment row

status=initiated, gateway='zarinpal', transaction_reference='A000000...' (the authority). No DB::transaction — external HTTP cannot be wrapped.

5
Response 200

{"type":"online","payment_id":2,"status":"initiated","redirect_url":"https://payment.zarinpal.com/pg/StartPay/A000000..."}

6
Client redirects user's browser to redirect_url

User pays at Zarinpal site. Flow continues with the callback below.

Online Payment Flow — Callback

1
GET /api/v1/payments/zarinpal/callback?Status=OK&Authority=A000000...

Zarinpal redirects user's browser here after payment attempt. No auth required.

2
PaymentController::zarinpalCallback()

Reads Status and Authority from query string. Guards against missing Authority (returns 400).

3
HandleZarinpalCallbackAction::handle('OK', 'A000000...')

Finds payment by transaction_reference = 'A000000...'.

4
Idempotency guard

If payment.status is already 'captured', return early with success — no double-processing.

5
ZarinpalGatewayDriver::verifyPayment(150000, 'A000000...')

HTTP POST to Zarinpal verify endpoint. Returns code=100, ref_id="123456789".

6
DB::transaction() — capture

Updates: status=captured, transaction_reference='123456789' (permanent ref_id), gateway_response={full JSON}. Calls orderManager->markAsPaid(42, '123456789'). Both writes in one transaction.

7
Response 200

{"success":true,"payment_id":2,"reference_id":"123456789"}

Failure paths

  • Status=NOK (user cancelled): Step 4 of callback action — writes status=failed, skips verify entirely, returns 422. Order stays pending. User can try again.
  • verifyPayment returns false: Step 6 fails the success check — writes status=failed + raw response, returns 422. Order stays pending.
  • Missing Authority: Controller guard catches it before the action is called, returns 400.

Why MockGatewayDriver instead of Http::fake()? Http::fake() intercepts at the HTTP client level and requires you to pattern-match URLs. The MockGatewayDriver approach tests the behavior your code exhibits when a gateway returns success or failure — the driver interface is the contract, the factory is the seam. This is far more readable and resilient to URL changes.

The Cents Rule

Every monetary value is stored and processed as a raw integer in the smallest currency unit (rials). No floats, no decimals, no conversion until the UI needs to format for display.

// CORRECT — integer rials everywhere
$amount = 150000;                                              // 150,000 rials
Payment::create(['amount' => $amount]);                      // stored as INT column
$driver->requestPayment(amountInCents: $amount);             // sent as-is to Zarinpal
$driver->verifyPayment(amountInCents: $payment->amount);    // read back, still int

// WRONG — never do this
$amount = 150000.00;                                          // forbidden float
$amount = $price / 100;                                       // creates a float

Enforced at three levels:

  1. DB column: $table->integer('amount') — rejects floats at the storage layer
  2. Eloquent cast: 'amount' => 'integer' — coerces on read from DB
  3. Source of truth: $order->totalAmount from OrderDTO is typed int — payment copies it directly with zero arithmetic
