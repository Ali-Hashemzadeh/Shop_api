01Ground rules that explain the shape of everything

Before the flows, five rules the code obeys everywhere. Once you know these, most of the "why is it written like that?" questions answer themselves.

Money

The Cents Rule

Every monetary value — item prices, shipping cost, totals — is a raw integer in rials. No floats, ever. A shipping price of 850,000 is the integer 850000.

Isolation

Contracts, not models

A module never imports another module's Eloquent model or joins its tables. They talk through interfaces (…ManagerInterface) resolved from the container, passing immutable DTOs.

Safety

Locks + idempotency

Anything that mutates shared state runs in a DB transaction with a row lock (lockForUpdate()), and every "make it happen" operation is safe to call twice.

History

Snapshots are frozen

Prices and the shipment selection are copied onto the order at checkout. Changing a product price or a config value later never rewrites an existing order.

Auth

Permission-based

Actions check a permission (order.create, shipment.post.hand-over), not a role. Guests get 401; authenticated-but-unauthorized get 403 before validation.

02The status vocabulary

Four separate state machines run in parallel. They are related but distinct — the same word can mean different things in different modules. Here is the full cast, colour-coded; the rest of the page reuses these colours.

Order status

The commercial summary a customer sees.

pending paid processing shipped completed cancelled failed

Payment status

Where a single payment attempt stands.

initiated pending_cash captured failed refunded

Shipment status

Operational fulfillment detail (per method).

pending preparing ready_for_post handed_to_post ready_for_dispatch out_for_delivery delivered delivery_failed ready_for_pickup picked_up cancelled

Slot reservation status

A seat in a local-delivery session.

held confirmed released expired cancelled completed
The one mapping to remember

The shipment status drives the order status after payment. Every fulfillment step calls back into Order to update its summary — preparing → order processing, handed_to_post → order shipped, delivered/picked_up → order completed.

03The happy path, end to end

Here is the entire lifecycle in one picture: a customer with a full cart checking out, paying, and the order being fulfilled. Every arrow is a real call across a contract. Read it top to bottom, then we'll unpack each module.

The single most important thing to notice: the operational shipment does not exist until the order is paid. At checkout, all that happens for shipping is a lightweight slot hold (and only for local delivery). Everything else waits for money.

04Order — the money and the record

The Order module turns a validated cart into an immutable financial contract. It stores two tables: orders (totals, status, the shipping-address snapshot, and now the shipment selection) and order_items (one snapshotted line per cart item — sku, title, quantity, and the price frozen at checkout).

Two columns were added for shipment:

ColumnPurpose
shipment_method_codeThe chosen method's stable code (e.g. local_delivery). This is the new source of truth.
shipment_snapshot (JSON)The whole selection frozen: method title/type, shipping cost, address, chosen slot, pickup location. Read an old order and you see exactly what was chosen then.
shipment_method_idLegacy integer column, kept nullable for backward compatibility. No longer written.

The checkout flow · CreateOrderAction

When POST /api/v1/orders arrives, the controller first asks Shipment to validate the selection (throwing a 422 with field errors on a bad address, ineligible region, or unbookable slot). It hands the resulting ShipmentSelectionDTO to the action, which does everything else inside a single transaction:

  1. Load the enriched cart and reject an empty one (422 EmptyCart).
  2. Compute totals — item subtotal + the shipping cost the selection carries. All integers.
  3. Release the previous pending order, if any. A user has at most one pending order; a new checkout replaces the old one, freeing its stock reservations and its held slot.
  4. Create the order as pending with shipment_method_code and the frozen shipment_snapshot.
  5. Snapshot each item and reserve its stock through InventoryManagerInterface.
  6. Hold the delivery slot — a no-op for postal/pickup; for local delivery it locks the slot row and creates a held reservation that expires when the order does.
  7. Clear the cart. Because this is the last step inside the transaction, if anything above fails the cart is left untouched.
Why the order matters

The cart is only cleared after stock is reserved and the slot is held. A failure anywhere rolls the whole transaction back — no half-checked-out state, no lost cart, no orphan reservation.

The shared "paid" path · markAsPaid

This is the hinge of the entire system. Both payment flows — in-person cash and online gateway — converge here. It is row-locked and idempotent:

  1. Lock the order and check its status. If it is already paid (or further along), return immediately — no double work.
  2. Flip to paid and store the transaction reference.
  3. Commit the inventory reservation for every item — turning "reserved" into "sold". This happens exactly once, guarded by the status check above.
  4. Activate the shipment via activateForPaidOrder() — which creates the operational record and confirms the held slot.
A gap this closed

Before shipment existed, paying an order never actually committed its inventory — reserved stock just sat reserved forever. Routing the commit through this one locked, idempotent path fixed that and guarantees it fires once.

Cancel & expiry

All the "unwind a pending order" paths share one primitive, CancelOrderAction::releaseAndCancel(): it releases every stock reservation, releases any held slot (via the Shipment contract), and marks the order cancelled. Three callers use it:

User

Cancel my order

POST /orders/{id}/cancel — owner-only, pending-only. 403 for someone else's order.

Scheduler

TTL sweep

orders:cancel-expired runs every minute, cancelling pending orders older than 15 minutes.

Checkout

Replace previous

Starting a new checkout releases the user's earlier pending order first.

05Payment — settling the order

Payment settles a pending order. It knows two method types and picks a gateway driver by name (the Strategy pattern: ZarinpalGatewayDriver in production, MockGatewayDriver in tests, resolved by PaymentGatewayFactory). Crucially, Payment never touches inventory or shipment directly — it only moves the order to paid through the contract, and markAsPaid does the rest.

Instant

In-person cash

initialize writes a pending_cash payment row and immediately calls markAsPaid. The order is paid the moment checkout-with-cash happens.

Two-step

Online gateway

initialize writes an initiated row and returns a redirect_url. The customer pays on Zarinpal; the public callback verifies, captures, and then calls markAsPaid.

Ownership guard

initialize aborts 403 unless the target order belongs to the caller — checked right after the 404 guard, so one user can never pay for (or, via cash, mark paid) another user's order.

The callback carries its own idempotency guard: if the payment is already captured, it returns early before doing anything. Combined with the lock inside markAsPaid, a payment gateway that fires its callback twice cannot double-commit inventory or create two shipments.

06Shipment — fulfilling the order

This is the largest module, but it decomposes into four clean concerns: the fixed methods, the local-delivery slot system, the operational shipment record, and the per-method workflows.

Config-backed methods — there is no methods table

The four fulfillment methods are fixed business facts, so they live in config/shipment.php, not the database. The admin cannot invent, rename, or reprice them. Each is keyed by a stable code and carries capability flags that drive the entire checkout UI.

CodeTypeAddressSlotTrackingEnds at
post_standardpostalrequired—yesshipment handed_to_post → order shipped
post_expresspostalrequired—yessame workflow as standard, different price/speed
local_deliverylocal_deliveryrequired (eligible)requirednoshipment delivered → order completed
in_person_pickuppickup——noshipment picked_up → order completed

validateSelection() is the checkout gate. Per method it enforces: is the code real and enabled? Is an address required and owned by the caller? For local delivery, is the address in an eligible region (a rule isolated behind LocalDeliveryEligibilityInterface) and is a bookable slot supplied? Is a slot prohibited (postal/pickup) but one was sent? Any failure is a 422 with the exact offending field.

The slot system — capacity like cinema seats

Local delivery works on dated sessions with limited capacity. Three layers build them:

  1. Working periods — recurring weekly templates (e.g. "Saturdays 15:00–22:00") in delivery_working_periods. These are the raw material; you seed them once.
  2. The generator — shipment:generate-delivery-slots slices each period into fixed-length sessions (default 90 min), keeping a short final slice only if it meets the minimum. It applies calendar exceptions (a date fully closed, or custom hours), and is idempotent: it never duplicates, overwrites an operator's edits, or reopens a closed slot.
  3. Dated slots — the concrete bookable rows in delivery_slots, each with a capacity.
Capacity is computed, never stored

There is no reserved_count column. Remaining capacity is derived live from the reservation table:

remaining = capacity − admin_reserved − active(held + confirmed)

A slot is selectable only if it is open, not in the past, past the minimum lead time, inside the booking horizon, not closed by an exception, and has remaining capacity > 0. The public availability endpoint returns full/closed slots as available: false so the UI can grey them out rather than hide them.

Overbooking is impossible

The availability list is only a snapshot. At checkout, holdForPendingOrder() re-locks the slot row with lockForUpdate() and re-counts capacity inside the transaction. Two customers racing for the last seat: one wins, the other gets a 422 on delivery_slot_id.

Activation happens on payment, and only once

When markAsPaid calls activateForPaidOrder(), the module: creates the shipments row (status pending, with the frozen snapshot copied in and an unguessable public_code), writes the first status-history entry, and confirms the held slot (held → confirmed). If the order carried no shipment selection, it simply returns nothing.

Idempotent by construction

The shipments.order_id column is unique. A repeated payment callback that reaches activation finds the existing shipment and returns it — no duplicate record, no duplicate history, no second slot confirmation.

Fulfillment workflows — one per method type

After activation, an operator moves the shipment forward through explicit business actions — there is no generic "set status" endpoint. Each transition is validated against a method-specific map by a ShipmentWorkflowResolver, and applied by one atomic primitive, ShipmentTransitionService, which locks the shipment, checks the move is legal, stamps the timestamp, writes a history row, and calls back into Order to update the summary status — all in one transaction. An illegal move (e.g. postal → out_for_delivery) returns a clean 422.

Postal · standard / express

Local delivery

Pickup

Postal tracking ends at handed_to_post

Once a parcel is handed to the postal service, the app has no reliable feed of the carrier's lifecycle — so it deliberately never claims in_transit, carrier out_for_delivery, or delivered for postal shipments. It shows the tracking number and tells the customer to follow it with the post office. handed_to_post is the terminal known state; the order sits at shipped.

Rescheduling a failed local delivery is its own guarded action: it locks and validates the new slot, releases the old reservation, creates a fresh confirmed one, updates the snapshot, and records the old→new slot in history — all without overbooking.

07How the guarantees are actually enforced

The properties the system promises aren't wishful — each is backed by a specific mechanism.

PromiseHow it's enforced
Inventory committed exactly once on paymentStatus check + row lock in markAsPaid: if already paid, return before committing.
No duplicate shipment on a repeated callbackUnique shipments.order_id + the callback's own "already captured" guard.
Two customers can't take the last slotlockForUpdate() on the slot row with a capacity re-check inside the transaction.
An abandoned order leaves no traceTTL sweep releases stock and slot; no shipment is ever created for an unpaid order.
Old orders are historically accurateEverything is read from the frozen shipment_snapshot, never recomputed from current config.
No circular dependency Order ↔ ShipmentOrder resolves the Shipment contract lazily (at call time) inside markAsPaid; the transition service injects the Order contract normally. The cycle never forms at construction.
