Shop API

Complete API reference for frontend developers. Every endpoint is documented with realistic request bodies and full response shapes.

Base URL

http://localhost:8000/api/v1

All paths in this document are relative to this base.

Content-Type

Send Content-Type: application/json for every JSON request.

File upload endpoints use multipart/form-data.

Authentication

Protected routes require: Authorization: Bearer {token}

Response Format

All responses are JSON. Paginated lists include data, links, and meta keys.

Overview

Quick Test (curl)

Copy-paste these to test the API locally while the server is running (php artisan serve):

# 1. Request OTP (creates user on first call)
curl -s -X POST http://localhost:8000/api/v1/otp/request \
  -H "Content-Type: application/json" \
  -d '{"phone":"09123456789","name":"Sara Ahmadi"}'

# 2. Verify OTP — replace CODE with the value logged by LogOtpSender
curl -s -X POST http://localhost:8000/api/v1/otp/verify \
  -H "Content-Type: application/json" \
  -d '{"phone":"09123456789","code":"CODE","device_name":"curl test"}'

# 3. Use the token from step 2
TOKEN="1|your_token_here"

# 4. Browse published products
curl -s http://localhost:8000/api/v1/catalog/products

# 5. Search with price ceiling
curl -s "http://localhost:8000/api/v1/catalog/products?search=گوشی&max_price=50000000"

# 6. Check stock for a SKU
curl -s http://localhost:8000/api/v1/inventory/sku/S25U-TBK-256

# 7. Upload an image (admin token required)
curl -s -X POST http://localhost:8000/api/v1/media \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@/path/to/image.webp" \
  -F "folder=products"
Finding the OTP in development: The app uses LogOtpSender which writes the code to the Laravel log. Run php artisan pail (or check storage/logs/laravel.log) and look for a line containing OTP and the phone number.

Authentication Flow

This API uses split-auth onboarding: new users verify phone ownership via OTP first, while returning users may choose OTP or a password. The flow is:

  1. Call POST /auth/check-user with the phone number. The response tells you whether this is a new account and which methods are allowed: a new phone returns {is_new_user: true, allowed_methods: ["otp"]}; an existing phone returns {is_new_user: false, allowed_methods: ["password", "otp"]}. Render the login UI accordingly.
  2. OTP path: Call POST /otp/request with the phone — the server sends an OTP code (SMS in production, logged in development). Then call POST /otp/verify with the phone, code, and a device name to receive a Bearer token. New users may also send name and password on verify to complete registration in one step.
  3. Password path (returning users only): Call POST /auth/login-password with the phone and password to receive a Bearer token directly.
  4. Store the token and include it as Authorization: Bearer {token} on every subsequent protected request.
  5. The first time a phone number is used, an account is created automatically — register and login are the same flow.
  6. After verifying, check user.name === null in the response. If null, show a profile completion screen and call PATCH /profile before continuing into the app.
is_new_user flag: POST /otp/request returns is_new_user: true when the phone was not previously registered. Use this to show different UI before the OTP step — e.g. "Create account" vs "Welcome back" — or to decide whether to show a profile completion screen after verification.
OTP codes are single-use and expire in 2 minutes. Calling /otp/request again generates a fresh code and invalidates the previous one. A wrong guess does not burn the valid code — it remains usable until expiry.

Errors & Validation

The API uses standard HTTP status codes:

  • 200 — OK
  • 201 — Created
  • 204 — No Content (successful delete)
  • 401 — Unauthenticated (missing or expired token)
  • 403 — Forbidden (token valid, but user lacks permission)
  • 404 — Not Found
  • 422 — Validation error

Validation errors (422) always look like this:

{
  "message": "The title field is required.",
  "errors": {
    "title": ["The title field is required."]
  }
}

Money & Prices

Important: All monetary values are stored and returned as plain integers in rials — the smallest currency unit. No floats, no currency symbols, no decimals. A product priced at 1,500,000 rials is returned as 1500000. Divide by 10 in your UI to display tomans.

Media Upload Strategy

The API supports two ways to attach images to any entity that accepts one:

  • Pre-upload (recommended): Upload the file first via POST /media, receive a media_id, then pass that integer ID to the create/update endpoint as a plain JSON field. This is the cleanest flow for image pickers.
  • Inline upload: Send the image file directly in a multipart/form-data request to the create/update endpoint itself. Convenient for simple single-step forms.
These two approaches are mutually exclusive per field. Inline upload is available on category, gallery, and standalone variant endpoints (send media_id OR the file field, never both). Product create/update do NOT accept inline files — pre-upload via POST /media and pass primary_media_id / gallery_media_ids / variants.*.media_id.

Every endpoint is protected by a named rate limiter. Exceeding the limit returns 429 Too Many Requests with a Retry-After header. Limits reset every 60 seconds.

Authentication — Login / Register

POST /auth/check-user Public 120/min · per IP

Resolves which authentication methods a phone number may use. Call this first to drive the login UI: new numbers must register via OTP (to prove phone ownership), while existing numbers can also use a password.

Request Body
FieldTypeDescription
phone_numberrequired11-digit mobile number starting with 09. Example: 09123456789
{
  "phone_number": "09123456789"
}
Response — 200 OK (new phone)
{
  "is_new_user":     true,
  "allowed_methods": ["otp"]
}
Response — 200 OK (existing phone)
{
  "is_new_user":     false,
  "allowed_methods": ["password", "otp"]
}
How to use: If allowed_methods contains "password", show a password field with a "Login with code instead" fallback. If it only contains "otp", send the user straight into the OTP flow.
POST /otp/request Public 5/min · per IP

Sends an OTP code to the given phone number. Creates the user account on first use. The same endpoint handles both first-time registration and subsequent logins.

Request Body
FieldTypeDescription
phonerequired11-digit mobile number starting with 09. Example: 09123456789
nameoptionalDisplay name. Only applied on account creation (first login). Ignored on return visits. Max 120 chars.
{

  "name":  "Sara Ahmadi"
}
Response — 200 OK
{
  "message":     "Verification code sent.",
  "expires_in":  120,   // seconds until the code expires (2 minutes)
  "is_new_user": true    // false if the phone already had an account
}
What to do with is_new_user: If true, the user was just created with only a phone number. After they verify the OTP, check user.name === null in the verify response and redirect to a profile completion screen.
POST /otp/verify Public 5/min · per IP

Verifies the OTP code and returns a Bearer token. The code is consumed immediately and cannot be reused. Store the returned token securely (e.g. localStorage or a secure cookie).

nameoptionalDisplay name set during first-time registration. Max 120 chars.

{

Catalog — Categories

Categories form an infinite tree. A category with parent_id: null is a root category. Children reference their parent via parent_id.

Every category response includes two tree fields:

  • parent — full ancestor chain going upward, each parent embedding its own parent, up to 5 levels. null for root categories.

Cart

The Cart module supports both guest and authenticated shoppers. All five endpoints use the cart.identify middleware — no 401 is ever returned for unauthenticated requests.

Session identity:
Authenticated users — send a Authorization: Bearer <token> header. The cart is permanently tied to your user ID.
Guests — send an X-Session-Id: <uuid> request header. If you omit it, a UUID is auto-generated and returned as X-Cart-Session-Id in the response. Store it and resend it on every subsequent request to keep the same cart.
All prices are integers (rials). base_price, compare_at_price, and line_total are raw integer rials. Divide by 10 for tomans. total_price on the cart object is the sum of all line totals.

Merge Guest Cart (Post-Login)

Merges a guest session cart into the authenticated user's permanent cart. Call this immediately after a successful login if the user had a guest cart.

Merge behaviour:
• SKUs that exist in both carts have their quantities summed, then clamped to available stock.
• SKUs only in the guest cart are moved to the user cart (quantity clamped to stock).
• SKUs with no inventory record are silently skipped — they don't block the merge.

Cart — Frontend Implementation Guide

This section walks through the complete frontend cart lifecycle from first visit to post-login merge. All examples use the native fetch API; adapt to Axios or any HTTP client as needed.

Step 1 — Guest browsing: create and persist a session

On a guest's first cart action, the server auto-generates a session ID and returns it as the X-Cart-Session-Id response header. Save it immediately — this is the guest's cart key for the entire browsing session.

async function guestCartRequest(method, path, body = null) {
  const sessionId = localStorage.getItem('cart_session_id');

  const headers = { 'Content-Type': 'application/json' };
  if (sessionId) headers['X-Session-Id'] = sessionId;

  const res = await fetch(`/api/v1/cart${path}`, {
    method,
    headers,
    body: body ? JSON.stringify(body) : undefined,
  });

  // Persist the session ID the server assigned (first request only)
  const returnedSession = res.headers.get('X-Cart-Session-Id');
  if (returnedSession) {
    localStorage.setItem('cart_session_id', returnedSession);
  }

  return res.json();
}

// Usage
await guestCartRequest('POST', '/items', { sku: 'S25U-TBK-256', quantity: 1 });
await guestCartRequest('GET',  '/');

Step 2 — Authenticated requests: use the Bearer token

Once the user is logged in, switch to sending the Authorization header instead of X-Session-Id. The server will always resolve the cart by user_id when a valid token is present.

async function authCartRequest(method, path, token, body = null) {
  const res = await fetch(`/api/v1/cart${path}`, {
    method,
    headers: {
      'Content-Type':  'application/json',
      'Authorization': `Bearer ${token}`,
    },
    body: body ? JSON.stringify(body) : undefined,
  });
  return res.json();
}

Step 3 — Post-login merge: combine the guest cart into the user cart

After a successful login, check whether a guest session existed. If it did, call the merge endpoint once, then clear the stored session ID. This is a fire-and-forget operation — if the guest cart is empty or the session ID is stale, the server handles it gracefully without error.

async function onLoginSuccess(token) {
  // Store the token however your app manages auth state
  localStorage.setItem('auth_token', token);

  const sessionId = localStorage.getItem('cart_session_id');

  if (sessionId) {
    await fetch('/api/v1/cart/merge', {
      method: 'POST',
      headers: {
        'Content-Type':  'application/json',
        'Authorization': `Bearer ${token}`,
      },
      body: JSON.stringify({ session_id: sessionId }),
    });

    // Guest session is no longer needed — the server deleted the guest cart
    localStorage.removeItem('cart_session_id');
  }

  // Reload the cart to reflect merged state
  return authCartRequest('GET', '/', token);
}
Do not merge on every page load. Only call /cart/merge once, immediately after the login response. Calling it later (e.g., on refresh) is harmless but wasteful — the guest cart will already be gone.

Step 4 — Logout: clear the auth token

On logout, remove the token from storage. Do not create a new guest session proactively — let the next cart action trigger it naturally via the X-Cart-Session-Id response header.

function onLogout() {
  localStorage.removeItem('auth_token');
  // Do NOT remove cart_session_id here — there is none yet for the new guest session.
  // The next cart action will create one automatically.
}

Orders

The Order module converts the authenticated user's current cart into an immutable financial record. Every order creation is atomic: prices are snapshotted, inventory is reserved, and the cart is cleared — all in a single database transaction.

Authentication required. All order endpoints require Authorization: Bearer <token>. Unauthenticated requests return 401. Users without the order.create permission return 403.
Price snapshot. price_per_unit and line_total on order items are fixed at the moment the order is created. They never change, even if the catalog price is updated later. All monetary values are raw integers (rials).
15-minute checkout TTL. A pending order that is not paid within 15 minutes is automatically cancelled and its inventory reservations are released. If the user tries to check out again, the old pending order is cancelled and replaced by the new one.
Auto-cancel on re-checkout. If a user places a new order while a pending order already exists, the old pending order is cancelled (reservations released) and a new order is created in the same request — no manual cleanup needed.
