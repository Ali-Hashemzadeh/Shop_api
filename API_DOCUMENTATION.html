Shop API

Complete API reference for frontend developers. Every endpoint is documented with realistic request bodies and full response shapes.

Base URL

http://localhost:8000/api/v1

All paths in this document are relative to this base.

Content-Type

Send Content-Type: application/json for every JSON request.

File upload endpoints use multipart/form-data.

Authentication

Protected routes require: Authorization: Bearer {token}

Response Format

All responses are JSON. Paginated lists include data, links, and meta keys.

Overview

Quick Test (curl)

Copy-paste these to test the API locally while the server is running (php artisan serve):

# 1. Request OTP (creates user on first call)
curl -s -X POST http://localhost:8000/api/v1/otp/request \
  -H "Content-Type: application/json" \
  -d '{"phone":"09123456789","name":"Sara Ahmadi"}'

# 2. Verify OTP — replace CODE with the value logged by LogOtpSender
curl -s -X POST http://localhost:8000/api/v1/otp/verify \
  -H "Content-Type: application/json" \
  -d '{"phone":"09123456789","code":"CODE","device_name":"curl test"}'

# 3. Use the token from step 2
TOKEN="1|your_token_here"

# 4. Browse published products
curl -s http://localhost:8000/api/v1/catalog/products

# 5. Search with price ceiling
curl -s "http://localhost:8000/api/v1/catalog/products?search=گوشی&max_price=50000000"

# 6. Check stock for a SKU
curl -s http://localhost:8000/api/v1/inventory/sku/S25U-TBK-256

# 7. Upload an image (admin token required)
curl -s -X POST http://localhost:8000/api/v1/media \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@/path/to/image.webp" \
  -F "folder=products"
Finding the OTP in development: The app uses LogOtpSender which writes the code to the Laravel log. Run php artisan pail (or check storage/logs/laravel.log) and look for a line containing OTP and the phone number.

Authentication Flow

This API uses split-auth onboarding: new users verify phone ownership via OTP first, while returning users may choose OTP or a password. The flow is:

  1. Call POST /auth/check-user with the phone number. The response tells you whether this is a new account and which methods are allowed: a new phone returns {is_new_user: true, allowed_methods: ["otp"]}; an existing phone that has set a password returns {is_new_user: false, allowed_methods: ["password", "otp"]}, while an existing phone with no password yet returns {is_new_user: false, allowed_methods: ["otp"]}. Render the login UI accordingly.
  2. OTP path: Call POST /otp/request with the phone — the server sends an OTP code (SMS in production, logged in development). Then call POST /otp/verify with the phone, code, and a device name to receive a Bearer token. Verify only proves phone ownership: capture the display name earlier via name/last_name on /otp/request, and add a password later via the authenticated POST /auth/set-password.
  3. Password path (returning users only): Call POST /auth/login-password with the phone and password to receive a Bearer token directly.
  4. Store the token and include it as Authorization: Bearer {token} on every subsequent protected request.
  5. The first time a phone number is used, an account is created automatically — register and login are the same flow.
  6. After verifying, check user.name === null in the response. If null, show a profile completion screen and call PATCH /profile before continuing into the app.
is_new_user flag: POST /otp/request returns is_new_user: true when the phone was not previously registered. Use this to show different UI before the OTP step — e.g. "Create account" vs "Welcome back" — or to decide whether to show a profile completion screen after verification.
OTP codes are single-use and expire in 2 minutes. Calling /otp/request again generates a fresh code and invalidates the previous one. A wrong guess does not burn the valid code — it remains usable until expiry.

Errors & Validation

The API uses standard HTTP status codes:

  • 200 — OK
  • 201 — Created
  • 204 — No Content (successful delete)
  • 401 — Unauthenticated (missing or expired token)
  • 403 — Forbidden (token valid, but user lacks permission)
  • 404 — Not Found
  • 422 — Validation error

Validation errors (422) always look like this:

{
  "message": "The title field is required.",
  "errors": {
    "title": ["The title field is required."]
  }
}

Money & Prices

Important: All monetary values are stored and returned as plain integers in rials — the smallest currency unit. No floats, no currency symbols, no decimals. A product priced at 1,500,000 rials is returned as 1500000. Divide by 10 in your UI to display tomans.

Media Upload Strategy

The API supports two ways to attach images to any entity that accepts one:

  • Pre-upload (recommended): Upload the file first via POST /media, receive a media_id, then pass that integer ID to the create/update endpoint as a plain JSON field. This is the cleanest flow for image pickers.
  • Inline upload: Send the image file directly in a multipart/form-data request to the create/update endpoint itself. Convenient for simple single-step forms.
These two approaches are mutually exclusive per field. Inline upload is available on category, gallery, and standalone variant endpoints (send media_id OR the file field, never both). Product create/update do NOT accept inline files — pre-upload via POST /media and pass primary_media_id / gallery_media_ids / variants.*.media_id.

Rate Limiting

Every endpoint is protected by a named rate limiter. Exceeding the limit returns 429 Too Many Requests with a Retry-After header. Limits reset every 60 seconds.

Authentication — Login / Register

POST /auth/check-user Public 120/min · per IP

Resolves which authentication methods a phone number may use. Call this first to drive the login UI: new numbers must register via OTP (to prove phone ownership), while existing numbers can also use a password.

Request Body
FieldTypeDescription
phone_numberrequired11-digit mobile number starting with 09. Example: 09123456789
{
  "phone_number": "09123456789"
}
Response — 200 OK (new phone)
{
  "is_new_user":     true,
  "allowed_methods": ["otp"]
}
Response — 200 OK (existing phone)
{
  "is_new_user":     false,
  "allowed_methods": ["password", "otp"]
}
How to use: If allowed_methods contains "password", show a password field with a "Login with code instead" fallback. If it only contains "otp", send the user straight into the OTP flow.
POST /otp/request Public 5/min · per IP

Sends an OTP code to the given phone number. Creates the user account on first use. The same endpoint handles both first-time registration and subsequent logins.

Request Body
FieldTypeDescription
phonerequired11-digit mobile number starting with 09. Example: 09123456789
nameoptionalFirst / display name. Only applied on account creation (first login). Ignored on return visits. Max 120 chars.
last_nameoptionalFamily name. Only applied on account creation (first login). Ignored on return visits. Max 120 chars.
{
  "phone":     "09123456789",
  "name":      "Sara",
  "last_name": "Ahmadi"
}
Response — 200 OK
{
  "message":     "Verification code sent.",
  "expires_in":  120,   // seconds until the code expires (2 minutes)
  "is_new_user": true    // false if the phone already had an account
}
What to do with is_new_user: If true, the user was just created with only a phone number. After they verify the OTP, check user.name === null in the verify response and redirect to a profile completion screen.
POST /otp/verify Public 5/min · per IP

Verifies the OTP code and returns a Bearer token. The code is consumed immediately and cannot be reused. Store the returned token securely (e.g. localStorage or a secure cookie).

Request Body

}

{

  "has_password": false,       // true if the account already has a password set

Catalog — Products

Products can be draft (admin-only) or published (public). Public routes only return published products. Each product response includes a nested images gallery array and a variants array (each variant carries a read-only stock count).

Read-only fields. A product's id is the short public code (a 7-char hex string, e.g. "a3f9c1b") used in every /products/{uuid} route — not the internal integer key. sales_count is a denormalized best-seller tally (units from realized orders), kept current by an hourly background sync. Both are server-computed and never accepted from client input.

The features field is an ordered array of {"title": "...", "value": "..."} objects — useful for displaying product specification tables.

Price filters (min_price / max_price) target the product's default variant (is_default: true) only, so the comparison reflects the storefront display price rather than any secondary variant.

Cart

The Cart module supports both guest and authenticated shoppers. All five endpoints use the cart.identify middleware — no 401 is ever returned for unauthenticated requests.

Cart — Frontend Implementation Guide

This section walks through the complete frontend cart lifecycle from first visit to post-login merge. All examples use the native fetch API; adapt to Axios or any HTTP client as needed.

Step 1 — Guest browsing: create and persist a session

On a guest's first cart action, the server auto-generates a session ID and returns it as the X-Cart-Session-Id response header. Save it immediately — this is the guest's cart key for the entire browsing session.

async function guestCartRequest(method, path, body = null) {
  const sessionId = localStorage.getItem('cart_session_id');

  const headers = { 'Content-Type': 'application/json' };
  if (sessionId) headers['X-Session-Id'] = sessionId;

  const res = await fetch(`/api/v1/cart${path}`, {
    method,
    headers,
    body: body ? JSON.stringify(body) : undefined,
  });

  // Persist the session ID the server assigned (first request only)
  const returnedSession = res.headers.get('X-Cart-Session-Id');
  if (returnedSession) {
    localStorage.setItem('cart_session_id', returnedSession);
  }

  return res.json();
}

// Usage
await guestCartRequest('POST', '/items', { sku: 'S25U-TBK-256', quantity: 1 });
await guestCartRequest('GET',  '/');

Step 2 — Authenticated requests: use the Bearer token

Once the user is logged in, switch to sending the Authorization header instead of X-Session-Id. The server will always resolve the cart by user_id when a valid token is present.

async function authCartRequest(method, path, token, body = null) {
  const res = await fetch(`/api/v1/cart${path}`, {
    method,
    headers: {
      'Content-Type':  'application/json',
      'Authorization': `Bearer ${token}`,
    },
    body: body ? JSON.stringify(body) : undefined,
  });
  return res.json();
}

Step 3 — Post-login merge: combine the guest cart into the user cart

After a successful login, check whether a guest session existed. If it did, call the merge endpoint once, then clear the stored session ID. This is a fire-and-forget operation — if the guest cart is empty or the session ID is stale, the server handles it gracefully without error.

async function onLoginSuccess(token) {
  // Store the token however your app manages auth state
  localStorage.setItem('auth_token', token);

  const sessionId = localStorage.getItem('cart_session_id');

  if (sessionId) {
    await fetch('/api/v1/cart/merge', {
      method: 'POST',
      headers: {
        'Content-Type':  'application/json',
        'Authorization': `Bearer ${token}`,
      },
      body: JSON.stringify({ session_id: sessionId }),
    });

    // Guest session is no longer needed — the server deleted the guest cart
    localStorage.removeItem('cart_session_id');
  }

  // Reload the cart to reflect merged state
  return authCartRequest('GET', '/', token);
}
Do not merge on every page load. Only call /cart/merge once, immediately after the login response. Calling it later (e.g., on refresh) is harmless but wasteful — the guest cart will already be gone.

Step 4 — Logout: clear the auth token

On logout, remove the token from storage. Do not create a new guest session proactively — let the next cart action trigger it naturally via the X-Cart-Session-Id response header.

function onLogout() {
  localStorage.removeItem('auth_token');
  // Do NOT remove cart_session_id here — there is none yet for the new guest session.
  // The next cart action will create one automatically.
}

Orders

The Order module converts the authenticated user's current cart into an immutable financial record. Every order creation is atomic: prices are snapshotted, inventory is reserved, and the cart is cleared — all in a single database transaction.

Authentication required. All order endpoints require Authorization: Bearer <token>. Unauthenticated requests return 401. Users without the order.create permission return 403.
Price snapshot. price_per_unit and line_total on order items are fixed at the moment the order is created. They never change, even if the catalog price is updated later. All monetary values are raw integers (rials).
Customer & product snapshots. Orders are historical records, so every order carries a frozen copy of the buyer's identity and each purchased product. customer_snapshot (name, last_name, phone, email) is captured on the order at checkout; each item's product_snapshot (title, sku, image_url, attributes) is captured on that order item. Neither is ever rewritten — if the customer later edits their profile, or an admin renames/re-images the product, this order still shows exactly what was true at checkout. Both fields are nullable JSON and appear on every order response (create, list, cancel).
15-minute checkout TTL. A pending order that is not paid within 15 minutes is automatically cancelled and its inventory reservations are released. If the user tries to check out again, the old pending order is cancelled and replaced by the new one.
Auto-cancel on re-checkout. If a user places a new order while a pending order already exists, the old pending order is cancelled (reservations released) and a new order is created in the same request — no manual cleanup needed.

Admin Orders — Frontend Implementation Guide

This is the complete guide for building the admin dashboard's Orders screens. Read it before wiring the UI — the most important thing to understand is what this module deliberately does not do, so you don't build buttons the API will never support. All examples use the native fetch API with an admin's Bearer token; adapt to your HTTP client.

The golden rule: the admin Orders screen is read-only except for one action — cancel. An admin can view, search/filter, and cancel a pending order. That's it. There is no endpoint to edit an order, create one on a customer's behalf, or change its status (mark-paid, mark-shipped, mark-complete, etc.). Do not build those controls — they will always 404/405.

Where order status actually changes (and why it's not here)

An order's status is a summary that the backend keeps in sync automatically. It moves forward as a side effect of two things:

  1. Payment — when a payment succeeds, the order becomes paid and a shipment record is created (see Payments).
  2. Fulfillment — the operator drives the parcel through the Shipment admin workflow (start-preparing → hand-to-post / out-for-delivery / picked-up …). Each shipment transition automatically syncs the order's summary status (processing → shipped → completed).

So if your admin UI needs a "mark this order as shipped" button, that button belongs on the Shipment screen and calls a Shipment admin action — never an Order endpoint. The Order detail here simply reflects the resulting status.

Mental model: Payment → order paid → shipment activated → operator runs shipment actions → order status auto-syncs. The admin Orders screen sits at the end of that chain as a viewer, plus a cancel escape hatch for orders that never got paid.

Two permissions govern this screen. Fetch the admin's permissions once at login (from your auth/profile response) and use them to decide what to render:

Step 4 — Render the shipment status block

data.shipment is the order's live fulfillment status, resolved through the Shipment module. It is null until the order is paid — an unpaid/pending order has no shipment yet, so show an "Awaiting payment" placeholder.

function renderShipmentBlock(order) {
  const s = order.shipment;
  if (!s) return 'Awaiting payment — no shipment yet';

  // s = { public_code, status, status_label, method_code, method_type, tracking_number }

Postal tracking ends at handed_to_post. For postal orders the app deliberately stops reporting carrier progress after handoff — show the tracking_number and tell the operator to track via the postal service. (See the Shipment section.)

Step 5 — The Cancel action (the only mutation)

Cancel is allowed only while the order is pending (not yet paid). This is the escape hatch for orders that were placed but never paid — cancelling releases the reserved stock and any held delivery slot, and flips the order to cancelled. Show the button only when both conditions hold: the admin has order.cancel-admin and order.status === 'pending'.

async function cancelAdminOrder(token, orderId) {
  const res = await fetch(`/api/v1/admin/orders/${orderId}/cancel`, {
    method: 'POST',
    headers: { Authorization: `Bearer ${token}` },
  });

  if (res.status === 403) throw { type: 'forbidden' };        // missing order.cancel-admin
  if (res.status === 404) throw { type: 'not_found' };
  if (res.status === 422) throw { type: 'not_pending' };      // someone paid it first — refresh the page
  const { data } = await res.json();
  return data; // the same detail shape, now status: "cancelled"
}
Handle the 422 race. Between loading the detail and clicking Cancel, the order may have been paid (its status left pending). The server rejects the cancel with 422 — don't treat this as a fatal error. Reload the order and show its new status; the Cancel button should now be gone.

Always confirm before cancelling (it releases inventory and cannot be undone), then re-render the page from the returned data (or reload) so the status, the vanished Cancel button, and the released stock are all reflected.

Shipment — Frontend Implementation Guide

This walks the storefront checkout UI through choosing a fulfillment method and (for local delivery) a delivery session, placing the order, and rendering the shipment status afterwards. All examples use fetch with a Bearer token; adapt to your HTTP client. Never send prices, dates, or slot times — the server resolves them from config and the snapshot; the UI submits only ids and the method code.

Step 1 — Let the user pick an address, then load methods

Call GET /shipment/methods with the selected address_id (omit it if none is chosen yet). Each method returns capability flags so you can drive the UI without hard-coding anything. Local delivery comes back available: false + unavailable_reason when the address is missing or ineligible — render it disabled with the reason.

async function loadMethods(token, addressId) {
  const q = addressId ? `?address_id=${addressId}` : '';
  const { data } = await (await fetch(`/api/v1/shipment/methods${q}`, {
    headers: { Authorization: `Bearer ${token}` },
  })).json();

  return data; // [{ code, title, type, price, requires_address, requires_delivery_slot, available, unavailable_reason, pickup_location? }]
}

// Render: title + formatted price; disable when !available and show unavailable_reason.
// Use the flags to decide which extra inputs to show (see Step 2).
Drive the form entirely from the flags: show the address picker when requires_address, show the slot picker when requires_delivery_slot, and show pickup_location for the pickup method. Do not branch on the code string beyond that.

Step 2 — If the chosen method needs a delivery slot, load sessions

Only local_delivery has requires_delivery_slot: true. Fetch bookable sessions for the address and render them grouped by date. Slots with available: false (full, closed, past, or outside the booking window) must be shown disabled, not hidden — use remaining_capacity to label a "Full" state.

  const { data } = await (await fetch(`/api/v1/shipment/delivery-slots${q}`, {
    headers: { Authorization: `Bearer ${token}` },
  })).json();

  return data; // [{ date: '2026-07-15', slots: [{ id, starts_at, ends_at, remaining_capacity, available }] }]
}

Step 3 — Build the checkout payload per method

The order endpoint is the same for every method; only the fields differ. Send the method code, plus the ids the flags told you to collect.

function buildCheckoutBody(method, { addressId, deliverySlotId, notes }) {
  const body = { shipment_method_code: method.code };
  if (method.requires_address)       body.address_id = addressId;
  if (method.requires_delivery_slot) body.delivery_slot_id = deliverySlotId;
  if (notes)                         body.notes = notes;
  return body;
}

// post_standard / post_express → { shipment_method_code, address_id }
// local_delivery             → { shipment_method_code, address_id, delivery_slot_id }
// in_person_pickup           → { shipment_method_code }

Step 4 — Place the order and handle the "slot just filled" race

The availability list is a snapshot; between loading it and checking out, the last place in a slot can be taken. The server re-checks under a lock and returns 422 with errors.delivery_slot_id. Handle it by reloading slots and asking the user to pick again — don't treat it as a fatal error.

async function placeOrder(token, body) {
  const res = await fetch('/api/v1/orders', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
    body: JSON.stringify(body),
  });

  if (res.status === 422) {
    const { errors } = await res.json();
    if (errors?.delivery_slot_id) {
      // Slot no longer available (or invalid) — refresh the slot picker.
      throw { type: 'slot_unavailable', errors };
    }
    throw { type: 'validation', errors }; // e.g. address_id ineligible, empty cart
  }

  return res.json(); // 201 → the order, including shipment_method_code + shipment_snapshot
}
Don't cache the slot list across the payment step. If the user backs out and returns, reload /shipment/delivery-slots — capacity and availability change constantly.

Step 5 — After payment, read and render the shipment

A shipment record exists only after the order is paid (see Payments). Then read it by the order id or the shipment's public code and render a status timeline from status + history.

async function loadShipment(token, orderId) {
  const res = await fetch(`/api/v1/orders/${orderId}/shipment`, {
    headers: { Authorization: `Bearer ${token}` },
  });
  if (res.status === 404) return null; // not paid yet → show "awaiting payment"
  return res.json(); // { id: 'SH-...', status, status_label, tracking_number, delivery_slot, pickup_location, history }
}

Variant per-order quantity limits

Every product variant may configure max_quantity_per_order. It limits that exact SKU inside one cart and resulting Order. null means no special Catalog limit; configured values must be whole integers of at least 1. Previous, paid, cancelled, pending, or other Orders are never counted. Inventory stock is independent and may impose a lower maximum.

Variant create and update

The field is accepted by standalone variant create/update, nested variants[] during Product creation, and variant upsert during Product update. Explicit null clears it. Existing Catalog permissions apply.

{ "sku": "bdp12-v3", "base_price": 125000, "max_quantity_per_order": 5 }
InputResult
null or omittedNo special per-order limit

0, negative, float, decimal string422 validation error

Product and variant responses

{ "sku": "bdp12-v3", "stock": 18, "max_quantity_per_order": 5, "effective_max_quantity": 5 }

effective_max_quantity is the lower of stock and the configured limit, or current stock when the limit is null. Response values are advisory; mutations reload authoritative Catalog and Inventory state.

Cart add, update, merge, and item response

POST /api/v1/cart/items validates existing + incoming quantity. PATCH /api/v1/cart/items/{itemId} validates the submitted final quantity. Guest and authenticated carts reject excess with 422 and remain unchanged.

{ "message": "The maximum allowed quantity for SKU bdp12-v3 is 5 per order.", "errors": { "quantity": ["You can add a maximum of 5 units of this item to one order."] } }

POST /api/v1/cart/merge sums duplicate SKUs and clamps to the lower of combined quantity, available stock, and the configured limit. Its response shape is unchanged.

{ "sku": "bdp12-v3", "quantity": 3, "available_stock": 18, "max_quantity_per_order": 5, "effective_max_quantity": 5, "remaining_addable_quantity": 2, "quantity_valid": true }

Lowering a limit does not rewrite existing carts; reads return quantity_valid: false until the customer reduces the quantity.

Order checkout and snapshot

POST /api/v1/orders aggregates all lines by SKU and reloads current limits before any Order mutation, Inventory reservation, local-delivery hold, or Cart clearing. A violation returns 422 and changes nothing.

{ "message": "The given data was invalid.", "errors": { "items.bdp12-v3.quantity": ["This item is limited to 5 units per order. Your cart currently contains 8."] } }

Order items expose nullable max_quantity_per_order_snapshot. It records the rule at checkout; later Catalog changes do not alter historical Orders and the snapshot never validates future Orders.
