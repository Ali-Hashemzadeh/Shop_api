Notifications & SMS

How a paid order turns into a message on a customer's phone — and why the code is arranged the way it is.

What This Is

Two modules that together answer one question: "something happened in the shop — who needs to know, and how?"

Modules/Notification

Stores in-app notifications, exposes the customer API, and fans a single request out across delivery channels. It knows how to notify, never when.

Modules/Sms

Picks the active SMS provider and translates our internal message shape into that provider's API format. It has never heard of orders, payments, or shipments.

Everything else — Order, Payment, Shipment — simply announces facts. They have no idea notifications exist.

Why Events Instead of Direct Calls

The obvious implementation is to call the notification manager straight from the payment callback. We deliberately don't, for three reasons:

  • Dependency direction. If Order called Notification, Order would depend on Notification — and on SMS by extension. With events, the arrow points the other way: Notification depends on Order's published event, a class with three integers in it. Delete the Notification module tomorrow and Order still compiles.
  • One dispatch, many reactions. A future "loyalty points" or "analytics" module listens to the same OrderPaidEvent without anyone editing markAsPaid.
  • Transaction safety. Listeners can be deferred until commit. A direct call inside a transaction would send an SMS for an order that later rolls back — and you cannot un-send an SMS.
On the enclosure rules. The project's core rule says a module never listens to another module's internal events. These events are different: they live in Domain/Events/, carry primitives only, and are documented as public contract — the same status as an interface or a DTO. Internal events, if any are ever added, stay private.

The request DTO is the whole API

Everything a caller can express is one object:

new NotificationRequestDTO(
    userId:   $event->userId,
    type:     NotificationType::PAYMENT_SUCCESS->value,
    title:    'پرداخت موفق',
    message:  'پرداخت سفارش شما با موفقیت انجام شد.',
    data:     ['order_id' => $event->orderId],
    channels: [NotificationChannel::DATABASE, NotificationChannel::SMS],
    sms:      new SmsPayloadDTO(NotificationTemplate::PAYMENT_SUCCESS, ['OrderId' => $event->orderId]),
);

The module stores and delivers it. It does not decide the copy, the type, or which channels apply — that judgement belongs to the flow raising it, which is why it lives in the listener.

The customer-facing API

EndpointBehavior
GET /api/v1/notificationsPaginated, newest first, always scoped to the caller. Permission notification.view-own.
POST /api/v1/notifications/{id}/readStamps read_at; idempotent. Another user's notification → 403, nothing mutated. Permission notification.mark-read-own.
The API never leaks delivery internals. NotificationResource exposes exactly id, type, title, message, data, read_at, created_at. Provider names, provider references, and error strings stay in notification_deliveries, which has no HTTP surface at all.

The stable internal format

SmsMessageDTO is what we mean, never what a vendor wants:

{
  "receiver":   "09121234567",      // canonical local format
  "template":   "payment_success",  // our name, not a vendor id
  "parameters": { "OrderId": 123 } // our business names
}

The provider does the translating. SMS.ir wants this:

{
  "mobile":     "989121234567",   // provider converts 09… → 98…
  "templateId": 424242,            // resolved from config by template name
  "parameters": [{ "name": "OrderId", "value": "123" }]
}

A different vendor might want { phone_number, template_id, variables: {...} }. That difference lives entirely inside its driver class. Nothing upstream changes — not the listener, not the channel, not the manager.

This is not the OTP sender. Identity has its own OtpSenderInterface / SmsIrOtpSender for login codes. Same vendor, different responsibility, separate contract — deliberately not shared. Do not merge them: OTP is an authentication primitive with its own throttle, TTL, and failure semantics.

Why notification_id is nullable. Some notifications are SMS-only by design — "your order is being prepared" is an operational nudge, not something worth a permanent row in the customer's in-app list. There is no parent notification to attach the delivery to, but we still want the attempt audited.

Indexes: [user_id, created_at] drives the listing, [user_id, read_at] drives the unread count, [channel, status] makes "show me today's failures" cheap.

Channels

A channel knows how to deliver. Adding one means implementing an interface and adding a line to a map — nothing else in the system changes.

ChannelDoesWrites
databaseCreates the in-app row the customer readsnotifications
smsResolves the phone via Identity, builds an SmsMessageDTO, hands it to SmsManagerInterface, records the outcomenotification_deliveries

Order matters. SendNotificationAction always runs the database channel first, so external channels can attach their delivery rows to a real notification id. External channels are then dispatched in a try/catch — a channel that explodes is logged, never rethrown.

Email and push are intentionally absent. Adding them would be a new enum case plus a new class; nothing in the listeners or business modules would need to change.

Optional SMS & Skipping

The single most important operational property: SMS is never mandatory and never fatal.

payment_success  ──▶ config lookup: sms.providers.smsir.templates.payment_success
                        │
       configured ──────┤────── missing
            │                     │
            ▼                     ▼
      send via provider     skip: info log
            │               + skipped delivery row
            ▼               + no HTTP call
    sent / failed row       business flow completely unaffected

A shop that has configured three of six SMS templates simply sends three kinds of message. Nothing throws, nothing rolls back, and the missing ones are visible in the delivery audit as skipped with a reason.

Why "skipped" is its own status. If an unconfigured template were recorded as failed, the delivery table would be full of red that nobody should act on — and a genuine provider outage would be invisible in the noise. failed means "we tried and it didn't work"; that's the signal worth alerting on.

The Three Outcomes

ResultMeansTypical causeAction
successThe provider accepted the message—none
skippedNothing was attempted, and that's fineNo template id configured, no API key, customer has no phone on fileconfigure it if you want that message
failedA real attempt did not succeedProvider down, HTTP 5xx, message rejected, caller requested SMS with no payloadinvestigate

These map straight onto SmsResultDTO (success / skipped / failure) and DeliveryStatus (sent / skipped / failed).

Why shipment events live in one place

Shipment already funnels every status change through a single atomic primitive, ShipmentTransitionService::transition(). Hooking in there — one announce() call driven by a match on the new status — means every workflow (postal, local delivery, pickup) is covered without touching a single action class, and no new status was invented. handed_to_post and out_for_delivery both raise ShipmentSentEvent because the module already maps both to the order status shipped.

Registration

// NotificationServiceProvider::boot()
Event::listen(OrderPaidEvent::class, SendOrderPaidNotifications::class);
Event::listen(OrderCancelledEvent::class, SendOrderCancelledNotifications::class);
Event::listen(PaymentFailedEvent::class, SendPaymentFailedNotification::class);
// …plus the three shipment events

This is the only file where the two sides meet. Laravel's auto-discovery only scans app/Listeners, so module listeners are registered explicitly — which is better anyway: one file tells you the entire integration surface.

Timing & Idempotency

Two hard requirements, solved by two different mechanisms.

1. Never notify for work that rolled back

Every listener implements Illuminate\Contracts\Events\ShouldHandleEventsAfterCommit. Laravel's dispatcher then holds the listener until the outermost transaction commits. If the inventory commit throws, or the shipment activation fails, or an enclosing transaction rolls back, the listener simply never runs.

DB::transaction:
    lock order
    status → paid
    save transaction reference
    commit inventory reservation
    activate shipment
    Event::dispatch(OrderPaidEvent)   ← queued, not executed
COMMIT ─────────────────────────────▶ listener runs here
   or
ROLLBACK ───────────────────────────▶ listener never runs, nothing sent
This matters more than it looks. The payment callback wraps markAsPaid in its own transaction. Without after-commit handling, a failure after the inner transaction but before the outer commit would leave a customer holding an SMS for an order that no longer exists.

2. Never notify twice for the same fact

Payment gateways retry callbacks. markAsPaid was already idempotent — if the order is in a sold state it returns the existing DTO early. The event dispatch sits after that early return, so it is physically unreachable on a repeat call:

if (already paid) {
    return $this->toDTO($order);   ← second callback exits here
}

// …transition work…
Event::dispatch(new OrderPaidEvent(...));  ← only the real transition reaches this

No dedupe table, no "already notified" flag, no cache key. The idempotency of the business operation is the idempotency of the notification.

Full Walkthrough: A Customer Pays

  1. Shopper is redirected back from Zarinpal; the gateway hits GET /api/v1/payments/zarinpal/callback.
  2. HandleZarinpalCallbackAction verifies server-side, marks the payment captured, and calls OrderManagerInterface::markAsPaid(). The controller contains no notification code whatsoever.
  3. markAsPaid locks the order, sets paid, commits the inventory reservation, activates the shipment, and dispatches OrderPaidEvent(orderId, userId, totalAmount).
  4. Transactions commit. Laravel now runs SendOrderPaidNotifications.
  5. The listener builds two requests: the customer's (database + sms) and — for every id returned by IdentityManagerInterface::getAdminUserIds() — an admin one (database only).
  6. DatabaseChannel inserts the customer's row. The customer's next GET /notifications shows پرداخت موفق.
  7. SmsChannel resolves the phone through Identity's contract, builds SmsMessageDTO('09…', 'payment_success', ['OrderId' => 42]), and hands it to SmsManagerInterface.
  8. SmsManager reads config('sms.default'), asks the factory for that driver, and calls it. SmsIrProvider looks up the template id, converts the number to 98…, and POSTs.
  9. The result comes back as an SmsResultDTO; SmsChannel writes a sent / skipped / failed delivery row.
  10. The gateway callback renders its Blade result page, oblivious to all of the above.
Notice what never happened. No controller touched a notification. No business module imported an SMS class. No model crossed a module boundary. Any single step from 5 onward could fail without affecting the payment.

Common Changes

Add a new notification for an existing event

Edit that event's listener. Copy, type, and channels all live there — one file, no infrastructure change.

Notify on a new business moment

  1. Add an event class to the emitting module's Domain/Events/. Primitives only.
  2. Dispatch it from the application-layer action or manager that owns the transition — never a controller.
  3. Add a listener in Notification/Application/Listeners implementing ShouldHandleEventsAfterCommit.
  4. Register it in NotificationServiceProvider::boot().
  5. Add a NotificationType case (and a NotificationTemplate case + config key if it sends SMS).

Add a new SMS provider

  1. Implement SmsProviderInterface in Sms/Infrastructure/Drivers; do all format translation inside it.
  2. Add one entry to SmsProviderFactory's driver map.
  3. Add its config block (api key, endpoint, template ids) to config/sms.php.
  4. Set SMS_PROVIDER=yourprovider. Nothing else in the codebase changes.

Turn one SMS on or off

Set or clear its SMS_SMSIR_*_TEMPLATE_ID. Empty means that message is skipped — no code change, no deploy risk.

Rules & Boundaries

✗
Never call NotificationManagerInterface from a controller, a gateway callback, an API resource, or an unrelated service. Dispatch an event instead.
✗
Never call an SMS provider directly. Everything goes through SmsManagerInterface.
✗
Never put a model in an event. Ids, strings, and numbers only — an event may be serialized, queued, or replayed.
✗
Never reuse Identity's OtpSenderInterface for notifications, or vice versa.
✗
Never let a notification failure break a business flow. If you add a channel, it swallows and records its own errors.
✓
Dispatch from the application layer (action / manager) that owns the state transition, after the work succeeds.
✓
Implement ShouldHandleEventsAfterCommit on every listener.
✓
Keep business copy in listeners; keep machine keys in NotificationType / NotificationTemplate.
✓
Resolve customer identity through IdentityManagerInterface, never the User model.

Two habits worth keeping:

  • Http::preventStrayRequests() in every SMS-touching test. Any unfaked outbound call fails the suite, so no test can ever hit a real SMS API — including by accident, in CI, on someone else's credentials.
  • Use FakeSmsProvider, not mocks. Set config('sms.default') to fake, resolve the singleton, and assert on sent() / lastMessage(). It also has shouldFail and shouldSkip flags to exercise both failure modes.

Known Gaps & Deliberate Silence

These are decisions, not oversights. Each is a one-line change if the business wants it.

Not notifiedWhy
In-store pickup (picked_up)The customer is standing at the counter; "your order was delivered" is redundant.
TTL order expiry (orders:cancel-expired)Every abandoned checkout would send an SMS. Likely spam, so it's off pending a product decision.
Superseded pending order during checkoutThe shared releaseAndCancel primitive also retires a draft the customer never asked about. Dispatching there would notify people about orders they never knew existed — there's a test asserting it stays silent.
Email / push / marketingOut of scope by design. The channel abstraction accommodates them; nothing implements them.
Admin notifications beyond paid ordersOnly admin_order_paid exists today.
Delivery is synchronous. Listeners run in-process after commit, so a slow SMS provider adds latency to the request that triggered it. If that becomes a problem, the listeners can implement ShouldQueue alongside the after-commit contract — the architecture already accounts for it, and no other code would change.
